# OPENAI.md — Hospici AI Coding Rules for OpenAI Models

> **Purpose:** This file governs all code generated by OpenAI models (GPT-4o, o1, o3, GPT-4 Turbo) in the Hospici repository.
> Read this entire file before generating any code, schema, migration, or documentation.
> These rules take priority over any default OpenAI code generation patterns.

---

## TL;DR — Read This First

These are the five errors OpenAI models most commonly generate in this codebase. Memorize them before generating a single line:

1. **Wrong Valkey package:** Use `iovalkey`, not `iovalis`
2. **Wrong Drizzle config key:** Use `dialect: "postgresql"`, not `driver: "pg"`
3. **TypeBox AOT violation:** Never call `TypeCompiler.Compile()` inside a function body
4. **RLS injection risk:** Never string-interpolate values into SQL SET statements
5. **Missing RLS:** Every new table needs `location_id` + RLS policies

---

## Project Context

Hospici is a HIPAA-compliant, CMS-regulated hospice EHR built with:
- **TypeBox** as the single source of truth for all schemas
- **Fastify 5** with hook-based middleware
- **Drizzle ORM** with PostgreSQL 18 and Row-Level Security
- **Valkey 8** (`iovalkey` package) for sessions, caching, and job queues
- **BullMQ** for background jobs with Dead Letter Queues
- **Better Auth** for authentication
- **FHIR R4** with R6 transformation adapters

## Rule 2 — Drizzle Config

  schema: "./src/contexts/**/schemas/*.ts",
  out: "./database/migrations/drizzle",

  driver: "pg",                                   // deprecated
  dbCredentials: { connectionString: "..." },     // wrong key
});

## Rule 3 — TypeBox AOT Compilation

All `TypeCompiler.Compile()` calls happen at module initialization level, once per schema.
The central registry is `src/config/typebox-compiler.ts`.

```typescript
// ✅ Module level — runs once at startup
import { TypeCompiler } from "@sinclair/typebox/compiler";
import { PatientSchema } from "../contexts/clinical/schemas/patient.schema";

export const PatientValidator = TypeCompiler.Compile(PatientSchema);

// ✅ Using the validator in a route
fastify.post("/patients", {
  preValidation: async (req, reply) => {
    if (!Validators.Patient.Check(req.body)) {
      // Validators imported from typebox-compiler.ts
    }
  }
});

// ❌ WRONG — compiles on every HTTP request
app.post("/patients", (req, res) => {
  const v = TypeCompiler.Compile(PatientSchema); // NEVER DO THIS
});
```

## Rule 4 — RLS Context Injection

```typescript
import { sql } from "drizzle-orm";

// ✅ CORRECT — parameterized set_config
fastify.addHook("preHandler", async (request) => {
  await db.execute(sql`SELECT set_config('app.current_user_id', ${request.user.id}, true)`);
  await db.execute(sql`SELECT set_config('app.current_location_id', ${request.locationId}, true)`);
  await db.execute(sql`SELECT set_config('app.current_role', ${request.user.role}, true)`);
});

// ❌ WRONG — string interpolation, even with type-validated uuid
fastify.addHook("preHandler", async (request) => {
  await db.execute(`SET app.current_user_id = '${request.user.id}'`); // NEVER
});
```

## Rule 5 — Table Design

Every new table that serves user data:

```typescript
// ✅ CORRECT Drizzle table definition
export const encounters = pgTable("encounters", {
  id: uuid("id").primaryKey().defaultRandom(),
  patientId: uuid("patient_id").references(() => patients.id).notNull(),
  locationId: uuid("location_id").references(() => locations.id).notNull(), // ← REQUIRED for RLS
  status: varchar("status", { length: 50 }).notNull(),                       // ← promoted: queries
  data: jsonb("data").notNull(),                                              // ← JSONB: variable content
  createdAt: timestamp("created_at").defaultNow(),
  updatedAt: timestamp("updated_at").defaultNow(),
});

// ✅ CORRECT migration (must accompany the table)
// ALTER TABLE "encounters" ENABLE ROW LEVEL SECURITY;
// CREATE POLICY "encounters_location_read" ON "encounters"
//   FOR SELECT USING (location_id = current_setting('app.current_location_id')::uuid);
```

**Column promotion policy:** `location_id`, `patient_id`, `status`, `fhir_version`, date fields used in billing → native columns. Everything else → JSONB.

## Rule 6 — Transactions for Atomic Operations

```typescript
// ✅ Patient admission + election statement — must not be split
return await db.transaction(async (tx) => {
  const [patient] = await tx.insert(patients).values(patientData).returning();
  const [election] = await tx.insert(electionStatements).values({
    patientId: patient.id,
    electionDate: input.electionDate,
    locationId: input.locationId,
    status: "active",
  }).returning();
  await tx.insert(auditLogs).values(auditData);
  return { patient, election };
});
```

Required for: NOE + benefit period, claim + audit log, patient admission + election, any 2+ table writes in a single business operation.

## CMS Business Rules Reference

| Rule | Implementation |
|---|---|
| NOE 5-day deadline | `addBusinessDays(electionDate, 5)` — skips weekends + federal holidays |
| Friday edge case | Friday election + 5 biz days = next Friday (not 5 calendar days) |
| IDG 15-day meeting | Hard block — not a warning. Prevents care plan update when overdue |
| HHA aide supervision | Every 14 days per CMS §418.76. BullMQ job alerts 2 days early |
| Benefit periods | 90 / 90 / 60 days. F2F required for period 3+ within 30 days prior |
| Hospice cap year | November 1 – October 31 (not January 1 – December 31) |
| Cap alert threshold | 80% of aggregate cap amount triggers notification |

## BullMQ / DLQ Pattern

Every BullMQ queue must have a Dead Letter Queue:

```typescript
const queue = new Queue("claim-processing", {
  connection: valkey,
  defaultJobOptions: {
    attempts: 3,
    backoff: { type: "exponential", delay: 1000 },
    removeOnFail: false, // keep for DLQ inspection
  },
});

const dlq = new Queue("claim-processing-dlq", { connection: valkey });

const events = new QueueEvents("claim-processing", { connection: valkey });
events.on("failed", async ({ jobId, failedReason }) => {
  const job = await queue.getJob(jobId);
  if (job && job.attemptsMade >= (job.opts.attempts ?? 3)) {
    await dlq.add("dlq", { ...job.data, failedReason });
    await alertService.send({ severity: "critical", message: `Job ${jobId} failed: ${failedReason}` });
  }
});
```

## Prohibited Patterns — Hard Stops

| If you are about to generate this | Stop and use this instead |
|---|---|
| `"iovalis"` | `"iovalkey"` |
| `driver: "pg"` | `dialect: "postgresql"` |
| `TypeCompiler.Compile()` inside a function | Move to module level |
| Template literal with `${userId}` in SQL | Use `sql` tagged template |
| New table without `location_id` | Add `location_id` + RLS |
| Two table writes without `db.transaction()` | Wrap in transaction |
| `console.log()` | `fastify.log.info()` |
| Hardcoded URL, key, or password | `process.env.VAR_NAME` |
