## 1. Non-Negotiable Rules

These rules are absolute. Violations will fail CI.

### 1.1 Package Names

```typescript
// ✅ CORRECT
import Valkey from "iovalkey";

// ❌ WRONG — this package does not exist in this project
import Valkey from "iovalis";
```

### 1.2 Drizzle Configuration

```typescript
// ✅ CORRECT (drizzle-kit 0.20+)
export default defineConfig({
  dialect: "postgresql",
  dbCredentials: { url: process.env.DATABASE_URL! },
});

// ❌ WRONG — deprecated, breaks migration generation
export default defineConfig({
  driver: "pg",
  dbCredentials: { connectionString: process.env.DATABASE_URL! },
});
```

### 1.3 TypeBox AOT Compilation

`TypeCompiler.Compile()` must be called **exactly once per schema, at module level.**
Never inside a function, class method, request handler, or loop. Never compile the same schema twice.

```typescript
// ✅ CORRECT — compiled once at module level in typebox-compiler.ts
Validators = {
  Patient: TypeCompiler.Compile(PatientSchema),
  ...
}

// ❌ WRONG — re-compiles on every call, defeats AOT, wastes CPU
async function handleRequest(req) {
  const validator = TypeCompiler.Compile(PatientSchema); // NEVER
}

// ❌ WRONG — duplicate compilation (schema file also compiles PatientSchema)
// patient.schema.ts:  export const PatientValidator = TypeCompiler.Compile(PatientSchema);
// typebox-compiler.ts: Patient: TypeCompiler.Compile(PatientSchema);  ← duplicate
```

All production validators live in `src/config/typebox-compiler.ts` as entries in the `Validators` object.
Do **not** add `TypeCompiler.Compile()` calls to schema files — schemas export only the `TSchema` value and types.
Consumers import `Validators.Patient`, `Validators.User`, etc. from `typebox-compiler.ts`.

This rule applies equally to the **Frontend** (`packages/shared-types` validators compiled at module level in `src/lib/validators/`).

### 1.4 RLS Context Injection — Parameterized Only

```typescript
// ✅ CORRECT — uses sql tagged template (parameterized)
await db.execute(sql`SELECT set_config('app.current_user_id', ${userId}, true)`);
await db.execute(sql`SELECT set_config('app.current_location_id', ${locationId}, true)`);

// ❌ WRONG — string interpolation, SQL injection vector
await db.execute(`SET app.current_user_id = '${userId}'`);
await db.execute(`SET app.current_location_id = '${locationId}'`);
```

### 1.5 Every New Table Requires RLS

Every table that stores user data must have:
1. `location_id UUID REFERENCES locations(id) NOT NULL` column
2. `ALTER TABLE x ENABLE ROW LEVEL SECURITY;`
3. At minimum a `_location_read` policy

No migration that adds a user-data table is valid without these.

### 1.6 Transactions for Multi-Step Operations

Whenever two or more tables are written in a single business operation, wrap in `db.transaction()`. Examples: NOE + benefit period, patient admission + election statement, claim + audit log.

## 2. Architecture Principles

### 2.1 Schema-First

1. Write TypeBox schema first
2. Derive Drizzle table from it
3. Generate migration
4. Compile validator in `typebox-compiler.ts`
5. Wire route with `preValidation` hook

Never write a Drizzle table before its TypeBox schema exists.

### 2.2 Column Promotion Policy

| Field type | Storage |
|---|---|
| `location_id`, `patient_id`, `status`, `fhir_version` | Native column |
| Date fields used in billing or cap queries | Native column |
| FHIR narrative, assessment scores, ABAC attributes | JSONB |

See `DATABASE_ARCHITECTURE.md §Column Promotion Policy` for the full table.

### 2.3 Domain Boundaries

Contexts must not directly import from sibling contexts. Use:
- Shared Kernel (`src/contexts/shared-kernel/`) for value objects
- Domain events for cross-context communication
- `src/schemas/` for cross-domain TypeBox definitions (FHIR core, CMS types)

### 2.4 Fastify Hook Order

`onRequest` → `preParsing` → `preValidation` → `preHandler` → `preSerialization` → `onSend` → `onResponse`

RLS context injection always goes in `preHandler`. TypeBox validation always goes in `preValidation`.

### 2.5 Frontend Stack

The frontend is **TanStack Start (RC)** — not Next.js. Key implications for code generation:

| Concern | Correct | Wrong |
|---------|---------|-------|
| Framework | TanStack Start (`@tanstack/react-start`) | Next.js |
| Routing | `createFileRoute` / `createRootRouteWithContext` | App Router / `useRouter` from next |
| Server logic | `createServerFn` + `createMiddleware` | Server Actions / Route Handlers |
| Data loading | `loader` in route + `createServerFn` | `fetch` in Server Components |
| Config | `vite.config.ts` + `app.config.ts` | `next.config.ts` |
| Client env vars | `import.meta.env.VITE_*` | `process.env.NEXT_PUBLIC_*` |
| Server-only env vars | `process.env.*` (no prefix) | `process.env.*` (same — correct) |
| Dev server port | `localhost:5173` (Vite default) | `localhost:3000` or `localhost:3001` |
| Auth guard | `_authed.tsx` `beforeLoad` | `middleware.ts` |
| RSC | Not supported | Supported |

Never generate Next.js-specific imports, file conventions, or config in the Frontend package.

## 3. CMS Compliance Patterns

### 3.1 NOE — 5-Day Rule with Friday Edge Case

```typescript
// addBusinessDays() skips weekends AND US federal holidays
const deadline = addBusinessDays(electionDate, 5);
// Test: election on Friday 2026-03-06 → deadline is Friday 2026-03-13
```

Always test the Friday edge case. Never use a simple `+5 days` without business day logic.

### 3.2 IDG — 15-Day Rule Is a Hard Block

IDG meetings are not optional. The enforcement mechanism is a **hard block** — not a warning. The system must prevent care plan updates if an IDG meeting is overdue.

The frontend companion to this rule: when `updateCarePlanFn` returns `{ code: 'IDG_OVERDUE' }`, the UI must show a modal with only one action ("Schedule IDG Meeting") — no dismiss option.

### 3.3 HHA Aide Supervision — 14-Day Rule

Every hospice aide must be supervised in person or virtually every 14 days per 42 CFR §418.76. The `aideSupervision` job alerts 2 days before deadline and blocks aide visit documentation if supervision is overdue.

### 3.4 Hospice Cap Year

The cap year runs November 1 to October 31. The cap recalculation BullMQ job runs on November 2. Never use a January 1 calendar year for cap calculations.

### 3.5 F2F Requirement

Face-to-face encounters are required for period 3 recertifications and must be within 30 days prior to the recertification date. The `certification.schema.ts` state machine blocks signing without a valid `f2fDate` when `f2fRequired = true`.

## 4. Security Rules

### 4.1 PHI Handling

- All fields listed in `PhiFieldSchema` are encrypted at rest via pgcrypto
- PHI must never be logged to Pino — use the `redact` config in `logging.config.ts`
- PHI must never appear in error messages returned to the client
- Break-glass access requires minimum 20-character reason and expires in 4 hours
- **Frontend:** PHI must never be stored in `localStorage` or `sessionStorage`.

### 4.2 HIPAA Audit Log

Every operation that reads or modifies patient data must emit an audit log entry. Use `AuditService.log()` — do not write directly to the `audit_logs` table outside the service.

Audit logs are append-only. Never generate a migration that adds UPDATE or DELETE policies to `audit_logs`.

### 4.3 BAA Vendors

Before integrating any third-party service (email, eRx, logging, hosting), confirm a signed BAA exists with that vendor. The BAA list is maintained in `docs/compliance/baa-registry.md`.

### 4.4 Token Storage (Frontend)

- Access tokens: memory only (JS closure) — never `localStorage`, never `sessionStorage`
- Session / refresh tokens: httpOnly cookie set by Better Auth — JS cannot read it
- Socket.IO auth: memory-stored access token passed in `auth` option at connection time

## 5. Testing Requirements

Every PR must include:

- **Unit test** for any new TypeBox schema (valid + invalid cases)
- **Integration test** for any new route
- **Edge case tests** for CMS business logic (Friday NOE, cap year boundary, F2F block)
- **RLS test** for any new table (`tests/validation/rls/`)
- **Frontend contract test** for any new server function (`tests/contract/`)

Test files live alongside their source: `patient.schema.ts` → `patient.schema.test.ts`

## 6. Code Style

- TypeScript strict mode, `verbatimModuleSyntax`
- No `any` — use `Type.Unknown()` in TypeBox or `unknown` in TypeScript
- No barrel `index.ts` files that re-export everything — import from the source file directly
- Prefer `const` over `let`; never `var`
- All async functions must handle errors — no unhandled promise rejections
- Prefer `Type.Enum({ key: 'value' })` over TypeScript enums for TypeBox schemas

**Frontend-specific:**
- Server-only code uses `.server.ts` suffix — never imported from client components
- `createServerFn` wrappers use `.functions.ts` suffix — safe to import anywhere
- Never import from `../../../backend/src/...` — use `@hospici/shared-types` workspace package

## 7. Migration Rules

1. Every migration file name must match: `XXXX_descriptive_snake_case.sql`
2. Get the next number via `npm run db:next-migration-number` — never guess
3. Every migration that adds a table must add RLS policies in the same file
4. Down migrations (rollback SQL) must exist for every migration
5. Never modify an existing migration file — create a new one

## 8. What Claude Must Never Do in This Repository

- Generate `import Valkey from "iovalis"` (wrong package)
- Generate `driver: "pg"` in any Drizzle config
- Generate `TypeCompiler.Compile()` inside a function or class method
- Generate string-interpolated `SET` statements for RLS context
- Generate a table migration without RLS policies
- Generate a multi-table write without `db.transaction()`
- Generate `console.log()` — use `fastify.log` or the Pino logger
- Generate hardcoded secrets, connection strings, or API keys
- Generate `UPDATE` or `DELETE` policies for `audit_logs`
- Skip the column promotion policy when designing new tables
- Generate Next.js imports, App Router conventions, or `next.config.ts` for the Frontend package
- Generate `process.env.NEXT_PUBLIC_*` — use `import.meta.env.VITE_*` in the Frontend package
- Store access tokens in `localStorage` or `sessionStorage`
- Import backend schemas directly into frontend files — use `@hospici/shared-types`
